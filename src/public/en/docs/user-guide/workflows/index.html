Cloud2Gether workflows let you automate repetitive tasks and create event-driven automations.

Workflow Types

Scheduled Workflows

Run tasks on a recurring schedule:

# daily-cleanup.yaml
name: Daily Cleanup
schedule: "0 2 * * *"  # 2 AM daily
agent: infrastructure
steps:
  - command: "Stop all EC2 instances tagged Environment=dev"
  - command: "Delete EBS snapshots older than 30 days"
  - command: "Remove unused Elastic IPs"

Event-Driven Workflows

Respond to infrastructure events:

# high-cpu-response.yaml
name: High CPU Response
trigger:
  type: cloudwatch
  metric: CPUUtilization
  threshold: 90
  duration: 5m
agent: infrastructure
steps:
  - command: "Analyze why {{ .resource }} has high CPU"
  - command: "If safe, scale horizontally by adding 2 instances"
  - notify:
      channel: "#ops-alerts"
      message: "High CPU detected on {{ .resource }}"

Manual Workflows

Pre-defined runbooks you trigger manually:

# deploy-production.yaml
name: Production Deployment
trigger: manual
agent: deployment
inputs:
  - name: version
    type: string
    required: true
  - name: rollback_on_failure
    type: boolean
    default: true
steps:
  - command: "Deploy {{ .inputs.version }} to production"
  - wait: 5m
  - command: "Verify deployment health"
  - on_failure:
      - command: "Rollback to previous version"
      - notify:
          channel: "#deployments"
          message: "Deployment failed, rolled back"

Creating Workflows

Using the CLI

Create a workflow from a YAML file:

c2g workflow create -f daily-cleanup.yaml

Or create interactively:

c2g workflow create --interactive

Using the Dashboard

  1. Navigate to Workflows in the sidebar
  2. Click Create Workflow
  3. Use the visual editor or YAML mode

Using Natural Language

Ask an agent to create a workflow:

> Create a workflow that runs every Monday at 9am and sends me a cost report

Created workflow: weekly-cost-report
Schedule: Every Monday at 9:00 AM
Agent: cost
Actions:
  1. Generate cost report for the past week
  2. Compare to previous week
  3. Send email to you@example.com

Workflow is now active.

Workflow Configuration

Schedule Syntax

Cloud2Gether uses standard cron expressions:

┌───────────── minute (0-59)
│ ┌───────────── hour (0-23)
│ │ ┌───────────── day of month (1-31)
│ │ │ ┌───────────── month (1-12)
│ │ │ │ ┌───────────── day of week (0-6, Sunday=0)
│ │ │ │ │
* * * * *

Common examples:

Schedule Cron Expression
Every hour 0 * * * *
Daily at midnight 0 0 * * *
Weekdays at 9 AM 0 9 * * 1-5
First of month 0 0 1 * *

Workflow Steps

Workflows support various step types:

steps:
  # Run an agent command
  - command: "List all stopped instances"

  # Wait for a duration
  - wait: 5m

  # Run steps in parallel
  - parallel:
      - command: "Check us-east-1"
      - command: "Check us-west-2"
      - command: "Check eu-west-1"

  # Conditional execution
  - if: "{{ .previous.result.count > 0 }}"
    then:
      - command: "Send alert"
    else:
      - command: "Log all clear"

  # Error handling
  - command: "Risky operation"
    on_failure:
      - notify:
          channel: "#alerts"
          message: "Operation failed: {{ .error }}"

Notifications

Send notifications to various channels:

- notify:
    channel: "#ops"           # Slack channel
    message: "Task completed"

- notify:
    type: email
    to: ["team@example.com"]
    subject: "Weekly Report"
    body: "{{ .report }}"

- notify:
    type: pagerduty
    severity: high
    message: "Critical issue detected"

c2g workflow pause daily-cleanup

c2g workflow run deploy-production --input version=v2.3.1

Workflow Examples

Cost Optimization Workflow

name: Weekly Cost Optimization
schedule: "0 9 * * 1"  # Monday 9 AM
agent: cost
steps:
  - command: "Find all unused resources"
    output: unused

  - command: "Calculate potential savings from {{ .unused }}"
    output: savings

  - if: "{{ .savings.total > 100 }}"
    then:
      - notify:
          channel: "#cost-alerts"
          message: |
            💰 Weekly Cost Optimization Report

            Found ${{ .savings.total }}/month in potential savings:
            {{ range .unused }}
            • {{ .type }}: {{ .name }} - ${{ .monthly_cost }}
            {{ end }}

Incident Response Workflow

name: Database Incident Response
trigger:
  type: cloudwatch
  alarm: database-connection-errors
agent: infrastructure
steps:
  - notify:
      type: pagerduty
      severity: high
      message: "Database connection errors detected"

  - command: "Check database {{ .resource }} health"
    output: health

  - command: "List recent changes to database configuration"
    output: changes

  - command: "Gather relevant logs from the last 30 minutes"
    output: logs

  - notify:
      channel: "#incidents"
      message: |
        🚨 Database Incident
        Resource: {{ .resource }}
        Health: {{ .health }}
        Recent changes: {{ .changes }}

        Runbook: https://wiki.example.com/db-incident

Deployment Pipeline

name: Production Deployment
trigger: manual
inputs:
  - name: service
    type: string
    required: true
  - name: version
    type: string
    required: true
agent: deployment
steps:
  - command: "Deploy {{ .inputs.service }}:{{ .inputs.version }} to staging"

  - wait: 2m
    reason: "Waiting for staging deployment to stabilize"

  - command: "Run integration tests against staging"
    output: tests

  - if: "{{ .tests.passed }}"
    then:
      - command: "Deploy {{ .inputs.service }}:{{ .inputs.version }} to production with blue-green strategy"
      - notify:
          channel: "#deployments"
          message: "✅ Deployed {{ .inputs.service }}:{{ .inputs.version }} to production"
    else:
      - notify:
          channel: "#deployments"
          message: "❌ Deployment blocked: integration tests failed"
      - command: "Rollback staging to previous version"

Best Practices

  1. Start simple — Begin with basic workflows and add complexity as needed
  2. Add notifications — Always notify on failures
  3. Use dry runs — Test workflows with --dry-run before activating
  4. Version control — Store workflow YAML files in git
  5. Document purpose — Add descriptions to explain why workflows exist
